Chapter 10: Testing and Debugging

  • If you are writing your own JS code, it's a good idea to add the line 'use strict' in the first line of the JS. If you are working on someone else's JS code, add the directive to each of your code blocks. Or use ES6 modules.
  • Linting tools can be used to check if your code meets a style guide and can highlight poor programming practices in your code.
  • Feature detection is used like so: if(api.callYouWantToUse){}
  • UA strings are not recommended for feature detection (but still used by Google to discriminate against people who don't use their browser)
  • The alert() method is the traditional way to debug JS code
  • The console API is extremely useful and well supported, but not officially part of spec
  • The debugger keyword can be used to set breakpoints in most browsers
  • There are a variety of error objects, including Error, EvalError (depreciated), RangeError, ReferenceError, SyntaxError, TypeError, URIError, and InternalError.
  • The throw keyword throws an error; can technically be any object but best practice to throw an error object.
  • Exceptions can be handled with try/catch blocks.
  • Test cases should be written
  • Test-driven Development is when test cases are written before the application code is written. It's best practice but not usually done
  • Jest is a JS test framework built by Facebook. To test your code, build xxx.test.js files and write your functions using test('test title', () => {expect(apiCall()).toBe(result)}); [.toEqual() for arrays, can also do .not.toBe(value) and .toThrow() for errors]
  • To run the tests, go to the parent folder and run jest -c {}
  • Refactor your code after fixing failed tests; red-green-refactor programming
